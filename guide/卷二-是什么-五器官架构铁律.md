# 卷二 · 是什么 · 元宪法

> **核心命题**：架构不是"建议怎么搭"，是"哪些底线绝对不能破"。破一条，大脑迟早烂。

卷一让你内化了世界观——代谢 > 囤积，五器官分工。但世界观是软的，人会偷懒、会图方便、会在某个深夜想"就这一次破例"。这一卷把六条铁律刻死，让你**即使图方便也不会把大脑搭坏**。

> **核心反直觉**：约束不是限制你，是保护你。最自由的知识库，恰恰是规则最硬的那个——因为它不会在你最需要它时背叛你。

> 阅读姿态：把这一卷当"宪法"读，不是"指南"。指南可以选择性采纳，宪法是底线，破了就出系统性故障。

## 第 6 章 · 三条治理线：单一真相源 / 只读不可变 / 本地优先

> **核心问题**：整个系统最高、不可违背的三条律是什么？
> **核心命题**：单一真相源、Layer1 只读不可变、本地优先——这三条是其余一切规则的地基。

### 6.1 真实的开场：当"产物"想僭越成"真相"

实践中跑过一个真实的张力：飞书看板上的数据和 知识库 里的数据对不上。团队有人下意识想"那以飞书为准吧，飞书新"。如果当时真这么做了，灾难就来了——飞书是**运营窗口**（产物/镜像），不是真相源；它的数据是从 Wiki 同步出去再被人手改的，改了不回流。一旦"以飞书为准"，你就有了**两个真相源**，从此再也分不清谁对。

正确处理是：飞书与 Wiki 冲突，**永远以 Wiki 为准**；飞书发现的修正，回流 Wiki 再同步出去。这条铁律见 本书架构 的"唯一真相源"原则。

这个案例说明：**多真相源 = 没有真相源。** 系统里必须有且只有一个地方说了算。

### 6.2 第一性定义：三条治理线

1. **单一真相源（Single Source of Truth）**：知识库 是唯一权威。其余四器官的内容若与它冲突，一律以它为准；器官的产物有长期价值就回流 Wiki，绝不允许出现"第二个说了算的地方"。
2. **Layer1 只读不可变（Immutable Raw）**：源材料（sources/）一旦入库，AI 只读不写。要基于它产出知识，写到 Layer2（concepts/syntheses），不动原文。这保证溯源永远可信。
3. **本地优先（Local-first）**：核心真相源和检索都在本地跑，不依赖任何云服务存活。云（GitHub/NotebookLM/飞书）是器官，断了大脑照样活。

### 6.3 反例对照：破线 vs 守线

| | 🔴 破治理线 | 🟢 守治理线 |
|---|---|---|
| 多处改数据 | 飞书改一处、Wiki 改一处 → 两个真相打架 | 只在 Wiki 改，同步到飞书 → 永远一致 |
| 改源材料 | 嫌 raw 有错直接改原文 → 溯源断裂 | 原文不动，在概念页里注明修正 → 溯源可信 |
| 依赖云 | 知识全在某 SaaS → 它涨价/封号你全完 | 本地真相源 + 云镜像 → 云挂了大脑还在 |

第三条尤其重要：源文章反复强调 QMD 的价值之一就是**完全本地、免费**——企业级 RAG 烧钱在托管向量库和 embedding API，QMD 把这两件都本地化了。本地优先不是省钱，是**主权**：你的大脑不能被任何第三方掐脖子。

### 6.4 参考实现的深层细节：为什么 Wiki 而不是飞书做真相源

有人会问：飞书也能存知识，为什么不让飞书当真相源？因为真相源的首要服务对象是 **AI**（agent 要高频读它来代谢），而飞书是为**人**优化的协作界面。本书架构 的边界铁律说得很清楚：飞书"不是给 AI 优化的真相源，AI 读它效率低"。真相源要的是纯文本、可 grep、可向量化、可版本化——这正是 markdown + 本地文件系统的强项，也是 LLM Wiki 模式（Karpathy）选 markdown 的根本原因。

### 6.5 训练阶梯：把三条线焊进直觉

| 时间 | 训练目标 | 自检方法 |
|---|---|---|
| 第 1 周 | 明确指认你的真相源是哪个目录 | 能一句话说出"我的真相源 = X" |
| 第 1 月 | 任何数据修改只在真相源做 | 0 次"在镜像上改了不回流" |
| 第 3 月 | 守住 Layer1 只读，不手痒改原文 | 所有修正都在 Layer2 注明 |
| 第 6 月 | 验证本地优先：断网/断云后大脑仍可用 | 拔网线后仍能检索+读知识 |

### 6.6 一个反例的代价：merge 不是 overwrite

实践中踩过一个真实的持久化事故：某次自动化脚本因为云端 clone 偶发失败、fallback 到空数据，**直接覆盖**了云端的完整版，把一个 49 条的数据集冲成 2 条。教训：**持久化层写入必须 merge（按主键并集），绝不能 overwrite**——这是"单一真相源"在工程层的延伸，真相源的写入必须数学保证单调不减。卷七第 37 章会把这条上升为伦理。

> **挂回**：本书架构（唯一真相源 + 边界铁律）· LLM Wiki 模式（Karpathy）（为什么 markdown 做真相源）· 卷六第 32 章（持久化覆盖的真实翻车）· 卷七第 37 章（真相完整性伦理）
> **下一章预告**：第 7 章拆开"真相源"内部——它不是一坨笔记，是**三层**精密结构：raw 原料 / wiki 知识 / 人类洞察，每层权限不同。

## 第 7 章 · 大脑本体三层：raw → wiki 页 → 洞察

> **核心问题**：真相源内部怎么组织，才既能堆原料又能产知识？
> **核心命题**：三层严格分离——Layer1 不可变原料 / Layer2 AI 维护的知识 / Layer3 人类独占的洞察。

### 7.1 真实的开场：数千 个文件怎么没乱成一锅粥

一套成熟的知识库实践 有 数千个 markdown 文件、上百 MB 文字。这么大的体量没乱，靠的就是 LLM Wiki 模式（Karpathy）的三层架构——每个文件一进来就被分到明确的层，权限和职责跟着层走。新手最容易犯的错是"所有笔记平铺在一个文件夹"，半年后就成了无法导航的垃圾场。

三层一分，每一类内容知道自己该在哪、谁能改它、怎么被检索。这是"活体大脑"和"数字垃圾场"的结构性分水岭。

### 7.2 第一性定义：三层及其权限

| 层 | 目录 | 内容 | 谁能写 | 不可变性 |
|---|---|---|---|---|
| **Layer 1 原料** | `sources/` `artifacts/` | 文章/纪要/转写/录音稿 | 人投喂，AI 只读 | 不可变 |
| **Layer 2 知识** | `concepts/` `syntheses/` `entities/` `comparisons/` | AI 编译的结构化知识 | AI 创建+维护，人审阅 | 可演化 |
| **Layer 3 洞察** | `insights/` | 人类主观判断/复盘/立场 | 人独占，AI 永不碰 | 人类专属 |

一句话记：**原料是死的（不可变）、知识是活的（AI 代谢）、洞察是你的（AI 只读引用）。**

### 7.3 反例对照：分层 vs 不分层

| | 🔴 不分层（一锅粥） | 🟢 三层分离 |
|---|---|---|
| 原料安全 | AI 改了原文，溯源断了 | Layer1 只读，原文永远可信 |
| 知识维护 | 人手动维护交叉引用，累死 | Layer2 AI 自动维护 |
| 洞察归属 | AI "优化"了你的主观判断 | Layer3 人独占，AI 不碰 |
| 规模可扩展 | 几百文件就乱 | 9000+ 文件仍可导航 |

### 7.4 参考实现的深层细节：一份源材料能长出 5-15 个知识页

按 LLM Wiki 模式（Karpathy）的 ingest 操作，**一份源材料消化后通常触发 5-15 个 Layer2 页的创建或更新**：提到的人建 entity 页、提到的概念建 concept 页、跨源的洞见建 synthesis 页。这就是"消化"这一代谢环（卷一第 3 章）的字面机制。

还有个进阶设计——"本体透镜"（ontology lens）：同一份源材料可以被多个视角读出不同概念（同一通客户电话，既能从财富视角读、也能从创业视角读）。这避免了"把一个事实强塞进单一分类"的失真。你的大脑长大后会自然遇到这个，知道它合理就行。

### 7.5 训练阶梯：建立三层手感

| 时间 | 训练目标 | 自检方法 |
|---|---|---|
| 第 1 周 | 建好三层目录结构 | sources/ concepts/ insights/ 三层就位 |
| 第 1 月 | 投喂只进 Layer1，知识只在 Layer2 | 0 次写错层 |
| 第 3 月 | 开始写 Layer3 个人洞察，AI 不碰 | 有 ≥3 篇纯人类洞察 |
| 第 6 月 | 三层代谢顺畅，一份源能自动长 5-10 页 | ingest 一篇看 Layer2 增长 |

### 7.6 一个辨析：Schema 是第四个隐形层

LLM Wiki 模式（Karpathy）其实还有一层常被忽略——**Schema（配置/宪法文件）**，就是定义规则和工作流的那份文档（实践中是 `_schema.md` + `CLAUDE.md`）。它不是知识，是"知识怎么组织"的元规则。每个进入大脑的 AI 第一动作就是读它。你建大脑时，这份 schema 文件就是你的 Claude.md（卷四第 19 章会教你写）。

> **挂回**：LLM Wiki 模式（Karpathy）（三层 + ingest 5-15 页 + schema）· 本书架构（Layer1/2/3 + 敏感数据子树）· 卷四第 19 章（Claude.md/schema 怎么写）· 卷七第 36 章（Layer3 洞察 = 人类独占的伦理边界）
> **下一章预告**：第 8 章把卷一画过的"五器官分工"升级成铁律——每个器官**只干一件事且不重叠**，违反就是把检索器官当大脑用（就像我们真干过的那次装错目标）。

## 第 8 章 · 五器官分工铁律：一个真相源 + 四器官，不重叠

> **核心问题**：五个系统的职责边界，硬到什么程度？
> **核心命题**：每个器官只干一件事且不重叠；任何器官都不能僭越成第二真相源。

### 8.1 真实的开场：把检索器官接到错的大脑上

卷一第 5 章提过：本书写作时，QMD 第一次被装在了一个工作目录（数百篇）上，而不是真大脑 知识库（数千 篇）。检索器官本身没装错，错在**接到了错的本体上**。发现后必须新建一个指向 知识库的 collection 才纠偏。

这个真实翻车精确地暴露了分工铁律的意义：**QMD 只是检索器官，它没有"自己的知识"，它的全部价值来自挂在正确的真相源上。** 同理，你不能指望 GitHub 帮你检索、不能指望 NotebookLM 当日常搜索、不能指望飞书当真相源——接错器官，整个系统就拧巴。

### 8.2 第一性定义：五器官职责表（铁律版）

| 器官 | 唯一职责 | 铁律：绝不做 |
|---|---|---|
| 🧠 知识库 | 真相源 | 不做全文语义检索 |
| 🔍 QMD | 全 vault 混合检索 | 不存知识、不编译、不当真相源 |
| 🦴 GitHub | 持久化/版本/跨机 | 不做检索、不当真相源 |
| 🔬 NotebookLM | 多源深综+音频 | 不做日常检索、不当真相源 |
| 🪟 飞书 | 看板/协作/PII运营 | 不当真相源 |

注意：**四个器官的"绝不做"里都有"不当真相源"**——这是第 6 章单一真相源治理线在器官层的强制落实。

### 8.3 反例对照：器官错位的后果

| 你想让它兼职 | 后果 |
|---|---|
| 🔴 让 QMD 当大脑（存知识在索引里） | 索引是 ~/.cache 临时产物，会被 `qmd embed -f` 清空——知识全丢 |
| 🔴 让飞书当真相源 | 双真相源打架（第 6 章那个故事） |
| 🔴 让 NotebookLM 当日常检索 | 太重、是云、有延迟，且不是本地真相源 |
| 🔴 让 GitHub 当检索 | git grep 抓不到语义，且要联网 |
| 🟢 各司其职 | 每个场景都有最强器官，无冲突 |

### 8.4 参考实现的深层细节：五器官如何形成闭环而不串味

五器官不重叠，但要协同。靠的是 本书架构 的闭环数据流 + 一条铁律：**素材永远先进真相源，再向器官辐射；器官产物有长期价值就回流真相源。** 这样既让器官各尽其能，又不让任何器官攒出"第二份真相"。

举个具体的：NotebookLM 做完一次深度研究，产物不是留在 NotebookLM（那会变成第二真相源），而是**回写到 Wiki 的 sources/**，再走正常的消化代谢。器官只是"加工车间"，成品入库还是回大脑。

### 8.5 训练阶梯：建立器官分工反射

| 时间 | 训练目标 | 自检方法 |
|---|---|---|
| 第 1 周 | 默写五器官"唯一职责 + 绝不做" | 不看书能复述 |
| 第 1 月 | 遇到需求先判断"调哪个器官" | 不再用错（如用本体硬搜） |
| 第 3 月 | 守住"器官产物回流真相源" | NotebookLM/飞书产物都回流 Wiki |
| 第 6 月 | 五器官协同成闭环 | 一条知识走完摄入→…→持久化无串味 |

### 8.6 第二个案例：检索器官的"模型一致性"也是分工的一部分

本书还踩过一个细节坑：QMD 的检索模型换成中文 Qwen3（1024 维）后，常驻的 MCP 服务进程还缓存着旧模型（768 维），导致维度不匹配报错。这提醒：**器官内部的配置一致性也要守**——检索器官的模型、索引、查询必须用同一套，换了就要全链重建。卷四第 20 章会讲这个具体怎么处理。

> **挂回**：本书架构（五器官分工铁律 + 闭环）· 卷一第 5 章（QMD 装错目标）· 卷四第 20-23 章（每个器官怎么装）· A7 教训卡
> **下一章预告**：第 9 章深入一对最容易混淆的器官关系——真相源的 wiki 页 vs QMD 的全文检索，到底谁干什么？答案是"编译 vs 检索"的互补铁律。

## 第 9 章 · 编译 vs 检索：wiki 管骨架，RAG 管细节

> **核心问题**：有了 wiki 知识页，为什么还需要 QMD？它俩不重复吗？
> **核心命题**：wiki+index 管"已编译的骨架知识"，QMD 管"全 vault 的零散细节"——互补，不替代。

### 9.1 真实的开场：那个找不回来的调试技巧

源文章里最戳人的一个场景：你往大脑塞了一个 30 篇的课程笔记系列，消化后压成了 2 个 wiki 页（一个课程地图、一个方法清单）。三个月后你想找"第 8 课讲的那个具体调试技巧"——**它不在那 2 个 wiki 页里**（wiki 只存骨架），它在 raw 原文里，但 grep 搜不到（你不记得关键词），index 也帮不上（它按页类型分类，不按内容）。那个技巧，"跟没存过一样"。

这个场景说明：**wiki 编译知识时，主动丢掉了细节——这是设计取舍，不是 bug。** 要把细节找回来，必须有另一层覆盖全 vault（含未编译 raw）的检索。这正是 QMD 的活。

### 9.2 第一性定义：编译层 vs 检索层

| | 编译层（wiki + index） | 检索层（QMD / RAG） |
|---|---|---|
| 干什么 | 把重要知识编译成结构化骨架 | 在全 vault 找跟 query 最相关的零散片段 |
| 知识形态 | 已消化、有交叉引用、有综合 | 原始片段、含未编译 raw |
| 强项 | "把已知重要的东西组织好" | "在海量内容里找跟当前问题相关的" |
| 类比 | 一本写好的教科书目录 | 一个能翻遍所有草稿的搜索引擎 |

源文章一句话总结：**"一个管编译，一个管搜索，互补。RAG 不是要替代 wiki+index，两者解决的不是同一个问题。"**

### 9.3 反例对照：只有一层的下场

| | 🔴 只有编译层 | 🔴 只有检索层 | 🟢 两层互补 |
|---|---|---|---|
| 找编译过的核心知识 | ✅ | 片段散，缺结构 | ✅ 读 wiki 页 |
| 找 raw 里的原话细节 | ❌ 永久丢失 | ✅ | ✅ QMD 召回 |
| 跨类语义查询 | ❌ 按类型分帮不上 | ✅ 向量命中 | ✅ |
| 知识的结构化综合 | ✅ | ❌ 只有片段 | ✅ 读 synthesis |

光有编译层 = 丢细节；光有检索层 = 没结构。实践中实测：QMD 索引 知识库 后，搜"心态变化 独立开发"能跨类命中 raw 的 Clippings 原文（0.88 分），这是编译层永远做不到的。

### 9.4 参考实现的深层细节：混合检索三式

"检索层"内部也不是单一技术。源文章讲透了：**真正靠谱的检索是混合检索**——
- **BM25** 抓精确（搜 "v2.1.0" 就要那个确切版本号，向量会给你 v2.0/v1.9）
- **向量** 抓语义（搜"心态变化"能命中"独立开发的尽头是无聊"，一字不重复）
- **小模型重排（rerank）** 把两路结果按相关性重新排，挑最好的几条

QMD 把这套（query expansion → BM25+向量并行 → RRF 合并 → qwen3-reranker 重排）打包了，"以前要写几万行代码，现在自动装好"。这跟 Google 搜索和 Anthropic 内部检索是同一套路。卷三第 14 章会教你怎么用好这三式。

### 9.5 训练阶梯：建立"编译 vs 检索"分诊直觉

| 时间 | 训练目标 | 自检方法 |
|---|---|---|
| 第 1 周 | 分清"找骨架知识"走 wiki、"找零散细节"走 QMD | 能对一个问题说出该走哪层 |
| 第 1 月 | 三式检索都用过（精确/语义/混合） | 各用过 ≥3 次 |
| 第 3 月 | 形成"先 QMD 定位、再 wiki 读结构"的复合用法 | 一次查询自然两层配合 |
| 第 6 月 | 不再纠结"重不重复"，两层各尽其用 | 0 次"用 grep 全文读烧 token" |

### 9.6 一句话锚定：Karpathy 自己的判断标准

LLM Wiki 模式（Karpathy）的提出者 Karpathy 在 gist 里原话：**"wiki 还小的时候 index 文件索引就够，但 wiki 长大之后你需要真正的搜索——qmd 是个好选择。"** 编译层是地基，检索层是 wiki 长大后的必需器官。两者不是二选一，是先后接入。

> **挂回**：LLM Wiki 模式（Karpathy）（编译层 ingest/query）· 本书架构（编译 vs 检索互补铁律）· 卷三第 14 章（混合检索三式怎么用）· 卷四第 20 章（QMD 怎么装）

## 第 10 章 · 数据可靠性分层：facts 可信，raw 存疑

> **核心问题**：大脑里的数字，哪些能直接信，哪些要打问号？
> **核心命题**：facts（人工校正层）最高信任，raw（纪要/转写）默认存疑——纪要数字实测会被系统性放大。

### 10.1 真实的开场：被放大 4 倍的财务数字

实践中做过一个真实校验：把会议纪要、Get 笔记、语音转写里的财务数字，和人工校正后的真实数字对比——**纪要里的财务数字被系统性放大了约 4 倍**。原因很多：口头表达夸张、转写错误、上下文丢失、复述失真。

如果你把纪要数字当事实直接用，你的大脑会"自信地告诉你错的东西"。实践中的解法是物理分层：建 `facts/`（人工校正，最高信任）+ `transcripts/`（原始转写，标记 reliability:raw），检索默认走 facts，纪要数字默认存疑、必须交叉验证。

这个案例说明：**数据进了大脑不等于数据可信。** 信任度必须是数据的一等属性。

### 10.2 第一性定义：两层信任

| 层 | 内容 | 信任度 | 用法 |
|---|---|---|---|
| **facts/** | 人工校正过的事实 | 最高 | 可直接引用 |
| **raw/transcripts/** | 纪要/转写/录音稿 | 默认存疑 | 必须交叉验证后才信 |

检索权重也跟着分：实践中的检索通道（knowledge-运营者）给 facts 层 **+5 权重**（压过一切），给 raw 转写命中标 **×0.25** 并提示回 facts 交叉验证。

### 10.3 反例对照：不分层 vs 分层

| | 🔴 数字不分信任层 | 🟢 facts/raw 分层 |
|---|---|---|
| 纪要说"收入 4000 万" | 直接信，写进报告 | 标 raw 存疑，回 facts 核 → 实际 1000 万 |
| AI 引用数字 | 引用了放大 4 倍的假数 | 默认引 facts，raw 须交叉验 |
| 决策依据 | 基于失真数据拍板 | 基于校正事实拍板 |

### 10.4 参考实现的深层细节：这是 PII 知识库的命门

数据可靠性分层在普通知识库是"建议"，在**含 PII 的业务知识库（如客户/客户数据）是命门**。实践中的 PZ-Framework 业务 Pack（如某业务知识包）严格区分：facts 层是主体盖章的事实，transcripts 是原始纪要——retrieval 默认走 facts，纪要数字默认存疑。卷七第 38 章会把这条上升为对数据主体的伦理。

### 10.5 训练阶梯：建立"数字默认存疑"反射

| 时间 | 训练目标 | 自检方法 |
|---|---|---|
| 第 1 周 | 区分哪些来源是校正事实、哪些是原始转写 | 能标出每个来源的信任级 |
| 第 1 月 | 重要数字引用前回 facts 核对 | 0 次直接引用未核纪要数字 |
| 第 3 月 | 建立 facts/transcripts 分目录 | 物理分层就位 |
| 第 6 月 | 形成"数字默认存疑"反射 | 看到纪要数字本能问"核过吗" |

### 10.6 一个推广：自报告失实是头号风险

数字会失真，AI 的"自报告"同样会失真——这是实践中反复强调的**头号红线**：AI 说"我装好了/我跑通了/没问题"不等于真装好了。本书写作时就遇到过工具安装器"自报成功"实则失实的情况。处理原则：**负证据要换工具复验一次**（find 一次空结果别下结论"没装"，换 ls/tree 二次确认）。卷三第 15 章会把"验真"训成一项肌肉。

> **挂回**：本书架构（facts vs raw 分层 + PII）· 卷三第 15 章（验真能力）· 卷七第 38 章（数据主体伦理）· A7 教训卡（纪要放大 4 倍 / 自报告失实）

## 第 11 章 · PII 物理隔离：本地可问答，git 不带走

> **核心问题**：客户/客户的敏感数据进了大脑，怎么既能用又不泄露？
> **核心命题**：PII 数据本地可索引可问答，但永不推 GitHub——物理隔离，by-design。

### 11.1 真实的开场：QMD 不读 .gitignore 这件事，反而救了 PII

本书接入 QMD 索引真大脑时，遇到一个关键判断：业务 Pack（含 PII，如客户/敏感数据）在 知识库的 `.gitignore` 里被整子树排除（防 GitHub 泄露）。问题是——**QMD 索引会不会因为 .gitignore 而漏掉这些 PII？**

实测发现 QMD **不读 .gitignore**（用自己的排除表）。这恰好形成了完美的隔离：QMD 在**本地**正常索引 PII（你能本地语义问答），但 QMD 的索引存在 `~/.cache/qmd`，**永不推 GitHub**；而 GitHub 同步走 git，git 读 .gitignore，**自动排除 PII**。两条路各管各的，PII 本地可用、云端不带走。

这个案例说明：**隔离不是"不索引"，是"索引在本地、传输被阻断"。** 你既要用 PII（本地问答），又不能让它出门（不进 git/云）。

### 11.2 第一性定义：PII 物理隔离契约

> **本地可问答、git 不带走**（PZ-Framework 契约）：
> - PII 数据**物理隔离**在专门子树（如 `敏感数据子树/`），用整子树 .gitignore 排除
> - **本地**：QMD 索引它、你能语义问答（不读 .gitignore）
> - **云/git**：GitHub mirror 自动排除（读 .gitignore）、飞书走脱敏 bridge
> - 敏感级别是**脊柱**而非字段：整个子树要么全本地、要么全隔离，不靠字段级标记

### 11.3 反例对照：字段级 vs 子树级隔离

| | 🔴 字段级标记隔离 | 🟢 整子树物理隔离 |
|---|---|---|
| 漏网风险 | 某个字段忘标 → 泄露 | 整子树一行 .gitignore，零漏网 |
| 同步安全 | 要逐字段判断能不能推 | 整子树永不推，无需判断 |
| 检索 | 复杂的字段级权限 | 本地全索引，云端全排除 |
| 维护 | 字段多了管不过来 | 一个 Pack = 一个子树，清晰 |

实践中的原话经验：**"传统中台 vs AI 原生的差别 = 敏感级是字段还是脊柱。"** 字段级总有漏网，子树级物理隔离才彻底。

### 11.4 参考实现的深层细节：脱敏 bridge 而非裸同步

PII 要给团队/移动端看怎么办？不是把 PII 推上去，而是走**脱敏 bridge**：飞书侧只接收脱敏后的聚合/事实，原始 PII 永远留本地。实践中的飞书运营节点就是这么设计的——真相源（含 PII）在本地，飞书是营业窗口（只见脱敏视图），bridge 负责中间的脱敏同步。这也呼应第 8 章"飞书不当真相源"的器官铁律。

### 11.5 训练阶梯：建立 PII 隔离纪律

| 时间 | 训练目标 | 自检方法 |
|---|---|---|
| 第 1 周 | 识别你大脑里哪些是 PII | 能列出所有 PII 子树 |
| 第 1 月 | PII 子树整体 .gitignore 排除 | git status 看不到任何 PII |
| 第 3 月 | 验证本地能问答、git 不带走 | QMD 本地搜得到 / git 推不出去 |
| 第 6 月 | PII 给人看走脱敏 bridge | 0 次裸 PII 上云 |

### 11.6 收束：六铁律为什么是地基

PII 隔离是六铁律里最有外部后果的一条（泄露是法律/信任事故），但它和前五条同源：单一真相源（PII 也只一处）、Layer1 只读（PII 原始不可变）、本地优先（PII 本地主权）、器官分工（飞书不碰原始 PII）、facts 分层（PII 数字也分信任）。**六条铁律不是六个规则，是一个一致的世界观在不同切面的投影。**
