# 🚧 混合检索反例案例 · 边界在哪

> 配套《混合检索经典案例》——那篇讲 QMD 的强项；这篇讲它**反而帮倒忙的 5 种场景**，让你知道边界。
> 核心反直觉：**默认用 `qmd query` 不是万灵药**。识别 5 类反例，省你的钱和时间。

---

## 反例 ① 精确版本号被语义稀释（最经典）

**场景**：你要找笔记里"qmd v2.1.0 升级记录"那条。
```bash
# 🔴 错：用混合查询
qmd query "qmd v2.1.0 升级记录" -c brain
# 实际命中：v2.0、v1.9 的笔记也排前面——向量觉得它们"语义太像"，rerank 也没分清
```
**对的问法**：
```bash
# 🟢 对：用 BM25 精确
qmd search '"v2.1.0"' -c brain    # 加引号强制字面匹配
```
**边界规则**：**确切版本号/特殊数字/精确专名 → 永远走 search，别走 query**。源文章原话：「向量搜'qmd v2.1.0'可能给你 v2.0/v1.9，因为语义太像。」

## 反例 ② 私有/新术语的"语义盲区"

**场景**：你库里有个自创术语"<你的自创术语>"，想找相关页。
```bash
# 🔴 错：以为向量能理解
qmd vsearch "顺势而行的创业哲学" -c brain
# 实际：Qwen3 预训练里没见过"<你的自创术语>"作为术语，向量给你一堆"乘风/顺势"的鸡汤文，专门那个概念页反而排第3
```
**对的问法**：
```bash
# 🟢 对：用 BM25 字面命中专有术语
qmd search "<你的自创术语>" -c brain
```
**边界规则**：**你库里独创的术语 / 公共预训练里没见过的词 → 向量是盲的，必须 BM25**。这是为什么 `qmd query` 的混合模式给 lex 段两倍权重——它知道私有术语只能靠字面。

## 反例 ③ Structured Query 的 hyde 写偏 → 整个查询带沟里

**场景**：写文章前找"深度工作的反共识观点"。
```bash
# 🔴 错：hyde 段写成了 "正方观点"
qmd query $'intent: 找反共识观点\nhyde: 深度工作很重要,Cal Newport 论证了 4 小时心流的价值,所有知识工作者都应该实践它...' -c brain
# 实际：hyde 描述的是主流观点 → 召回全是支持深度工作的页 → 反共识观点一个没找到
```
**对的问法**：
```bash
# 🟢 对：hyde 真写"反共识答案"长什么样
qmd query $'intent: 找反共识观点\nhyde: 深度工作概念被过度神化,真实知识工作 80% 是协作与沟通而非长时间独处;Newport 假设的"作家式工作"不适用于大多数岗位...' -c brain
```
**边界规则**：**hyde 段必须写"你期望的命中长什么样"，不是"主流认知"**。hyde 是杀手锏也是反噬源——写偏一字，全盘错。

## 反例 ④ 小库 / 主题完全没存 → 混合检索是杀鸡用牛刀

**场景**：你库就 50 篇笔记，找一个主题。
```bash
# 🔴 错：默认用 query，等了 3 秒
qmd query "时间管理方法" -c brain
# 实际：query expansion + 向量 + rerank 跑了 3 秒，库里就 3 篇相关——直接 grep 0.1 秒搞定
```
**对的问法**：
```bash
# 🟢 对：小库 / 简单意图用 search
qmd search "时间管理" -c brain
# 或者更暴力：grep -r "时间管理" ~/vault/
```
**边界规则**：**库 < 100 篇 / 主题简单 / 你知道关键词 → BM25 即可，混合是浪费**。混合检索的价值在中大规模 + 模糊意图时显现。

**更狠的边界**：如果主题你库里**根本没存**，再混合也是空——这时该走深研器官（NotebookLM）或外网搜索，不是死磕本地库。

## 反例 ⑤ 跨语言混排 → BM25 抓中文、向量抓英文、rerank 选错

**场景**：搜中英混合的概念，比如 "deep work 的中国职场版本"。
```bash
# 🔴 错：直接混合
qmd query "deep work 的中国职场版本" -c brain
# 实际：BM25 命中纯英文页（"deep work"）、向量命中纯中文页（"专注力"），rerank 反复横跳选不准
```
**对的问法**：
```bash
# 🟢 对：拆成两次查
qmd search "deep work" -c brain        # 先抓英文锚定页
qmd vsearch "中国职场的深度工作 本土化" -c brain  # 再抓中文语义页
# 自己合并 / 在 hyde 里把中英都铺一遍
```
**边界规则**：**跨语言混排时，混合检索的 RRF 合并会卡顿**。要么分两次，要么 Structured Query 的 lex/vec/hyde 段都同时写双语。

## 边界总览表

| 场景 | 默认 query | 实际该用 | 为什么 |
|---|---|---|---|
| 确切版本号/专名/特殊数字 | 🔴 被稀释 | ✅ `search '"v2.1.0"'` | 向量觉得 v2.0/v1.9 太像 |
| 你库独创术语 | 🔴 向量盲 | ✅ `search "术语"` | 预训练里没见过 |
| hyde 写偏 | 🔴 全盘错 | ✅ 重写 hyde 段 | hyde 直接定义召回目标 |
| 库 < 100 篇 | 🔴 杀鸡牛刀 | ✅ `search` 或 grep | 没数据规模红利 |
| 跨语言混排 | 🔴 RRF 卡顿 | ✅ 分两次或双语 hyde | BM25/向量抓不同语言 |

## 自检清单（开 query 前 10 秒过一遍）

- [ ] 我要的是**确切词**吗？→ 走 search，不走 query
- [ ] 我用的是**独创术语**吗？→ 走 search
- [ ] 我的 hyde 段写的是**期望命中**还是**主流共识**？→ 必须是前者
- [ ] 我库**够大**且我**说不清关键词**吗？→ 这才是 query 的最佳战场
- [ ] 主题我库里**根本有存**吗？→ 没存的走深研器官，别死磕本地

5 个 ✅ 才用 query；任一 🔴 切到对应方案。

## 一句话记忆

> **混合检索不是万灵药。精确词走字面、私有术语走字面、hyde 写偏全盘错、小库不用 query、跨语言要分流。**
> **`qmd query` 是默认，不是只用它。**
