# 卷四 · 用什么 · 元工具

> **核心命题**：五器官的具体工具栈 + 真实踩坑——但工具是仆人，不是主人。

## 第 19 章 · 大脑本体：Obsidian + 知识库 schema + Claude.md

> **核心问题**：真相源用什么工具，怎么初始化？
> **核心命题**：Obsidian（载体）+ 三层 schema（结构）+ Claude.md（AI 规则）= 一个能自我组织的大脑本体。

### 19.1 真实的开场：一句话让 AI 把整套目录建好

源文章描述的真相源初始化只要五步：装 Claude Code、装 Obsidian、新建 Vault、放一份 Claude.md 规则（用 Karpathy 的 LLM Wiki gist），然后对 Claude Code 说一句：**"帮我按规则配置好这个知识库"——它会自己把整套目录建好**（raw/、wiki/、index.md、log.md）。

这就是 LLM Wiki 模式（Karpathy）的威力：你不用手动建结构，给 AI 一份 schema，它替你搭。一套成熟的知识库实践 就是这套的成熟版（三层 + operations + 敏感数据子树）。

### 19.2 第一性定义：大脑本体三件套

| 件 | 是什么 | 作用 |
|---|---|---|
| **Obsidian** | markdown 笔记软件 | 真相源的载体（人看的界面） |
| **三层 schema** | 目录结构 + frontmatter 规则 | 知识怎么分层组织（卷二第 7 章） |
| **Claude.md** | AI 行为规则文件 | 每个进大脑的 AI 第一动作就读它 |

注意：**Obsidian 只是载体，不是大脑**。大脑 = markdown 文件 + schema + AI 维护。换个编辑器（VS Code/Typora）也能用，因为底层是纯文本文件——这正是"本地优先"（卷二第 6 章）的好处。

### 19.3 反例对照：有 schema vs 没 schema

| | 🔴 没 schema（裸 Obsidian） | 🟢 有 schema + Claude.md |
|---|---|---|
| 结构 | 笔记平铺，半年成垃圾场 | 三层自动组织 |
| AI 行为 | 每次都要重新解释规则 | AI 读 Claude.md 就懂 |
| 可扩展 | 几百文件就乱 | 9000+ 文件仍可导航 |

### 19.4 参考实现的深层细节：Claude.md 该写什么

实践中的 Claude.md（AI 行为路由）包含：AI 入口压缩层（先读哪两份定位地图）、不可变层硬约束（哪些只读）、可维护层权限、子路由入口（各领域 MOC）、代谢管道清单、页面模板、操作规范。核心是**让任何 AI 进来都能在 1500 字内知道"我能做什么、不能动什么、去哪找什么"**。Karpathy 的 gist 是个好起点，实践中的 `_schema.md` + `CLAUDE.md` 是个成熟参考。

### 19.5 训练阶梯：搭大脑本体

| 时间 | 训练目标 | 自检方法 |
|---|---|---|
| 第 1 周 | 装 Obsidian + Claude Code + 放 Claude.md | "帮我配置"后三层目录就位 |
| 第 1 月 | 喂前 10 篇，验证 schema 自动组织 | Layer2 自动长出概念页 |
| 第 3 月 | 根据自己领域定制 Claude.md 子路由 | AI 进来定位准 |
| 第 6 月 | 加 AI 入口压缩层（_ai-entry/_hot） | AI 首读 1500 字就定位 |

### 19.6 一个进阶：Obsidian 的隐藏坑

实操提醒（本书写作时实测）：外部工具（如 AI）在 Obsidian 运行时创建的新文件/文件夹，Obsidian 有时不会立刻刷新出来，需要 `Cmd+R` 重载或重启。还有：Obsidian 的文件名搜索匹配的是**文件名/内容**，不是文件夹名——搜文件夹名会 0 结果。这些小坑知道了就不慌。

> **挂回**：LLM Wiki 模式（Karpathy）（一句话建目录 + schema）· 本书架构（三层 + AI 入口压缩）· 卷二第 7 章（三层架构）· 卷二第 6 章（本地优先）
> **下一章预告**：第 20 章是全书最实操的一章——检索器官 QMD 的完整安装，含中文必切 Qwen3、HF 镜像、config 优先级三个真实大坑。

## 第 20 章 · 检索器官：QMD（中文必切 Qwen3）

> **核心问题**：QMD 怎么装，中文知识库要注意什么？
> **核心命题**：六条命令装好，但中文必切 Qwen3-Embedding、国内必走 HF 镜像、改模型必须改 config 而非只设环境变量。

### 20.1 真实的开场：源文章的六行安装 prompt

源文章给的安装方法是"一个提示词搞定"——cd 到大脑目录，把这段粘给 Claude Code：

```
npm install -g @tobilu/qmd                              # 装 CLI 本体
claude plugin marketplace add tobi/qmd && claude plugin install qmd@qmd  # 装 plugin(MCP+skill)
qmd collection add . --name brain                       # 当前目录配为知识库
切到 Qwen3-Embedding-0.6B 多语言模型（中文知识库必做）
qmd pull                                                # 拉本地小模型
qmd embed -f                                            # 生成向量索引
```

第一次会下载约 2.4GB 本地模型，几分钟。**之后所有 embedding 都在本地跑，不联网、不烧 API。** Claude Code 调大脑时自动走 QMD，你无感。

但本书实际装的时候，这六行背后藏着三个大坑——下面逐个拆。

### 20.2 第一性定义：QMD 装好的四个组件

| 组件 | 作用 |
|---|---|
| CLI 本体 | `qmd query/search/vsearch` 命令 |
| plugin（MCP+skill） | Claude Code 里自动调用 |
| collection | 指向你的大脑目录的索引配置 |
| 本地模型（embedding/rerank/query-expansion） | 全本地跑，不烧 API |

### 20.3 反例对照：三个真实大坑

| 坑 | 🔴 不知道会怎样 | 🟢 正确做法 |
|---|---|---|
| **中文模型** | 用默认英文 embedding，中文检索质量差一截 | 切 `QMD_EMBED_MODEL=hf:Qwen/Qwen3-Embedding-0.6B-GGUF/...`（源文章强调"英文教程里没人讲"） |
| **国内下载** | 直连 HuggingFace 卡在"Gathering information"零下载 | 设 `HF_ENDPOINT=https://hf-mirror.com`，秒下 20MB/s |
| **改模型不生效** | 只设环境变量，QMD 仍用旧模型 | QMD 把模型写进了 `~/.config/qmd/index.yml`，config 优先级 > 环境变量，必须改 config |

第三个坑最隐蔽：本书写作时，明明设了 `QMD_EMBED_MODEL` 环境变量，QMD 还是用默认 embeddinggemma——因为它第一次运行时把默认模型**写进了 config 文件**，而 config 优先级高于环境变量。且 QMD 的 MCP 服务是 Claude Code 拉起的子进程，**根本不读你 shell 的环境变量**，只有改 config 文件才能 CLI + 应用内一致。

### 20.4 参考实现的深层细节：换模型必须全链重建

QMD 换 embedding 模型后有个连锁反应（本书实测）：不同模型的向量维度不同（embeddinggemma 768 维 vs Qwen3 1024 维），**换了模型必须 `qmd embed -f` 全量重建索引**（向量不跨模型通用），且要重启 Claude Code 让 MCP 服务重载新模型——否则常驻 MCP 进程缓存旧模型会报"维度不匹配"。本书实测 知识库 全量（数千文档/数千 chunks）用 Qwen3 重建只花了 30 分钟，比预估 3-4 小时快得多。

日常维护只要 `qmd update`（增量索引，秒级），不用每次全量。

### 20.5 训练阶梯：装好检索器官

| 时间 | 训练目标 | 自检方法 |
|---|---|---|
| 第 1 周 | 六行装好 QMD，切 Qwen3，embed 完成 | `qmd status` 看到 Vectors 数 |
| 第 1 月 | 三式检索都跑通中文 | 搜中文有合理命中 |
| 第 3 月 | 挂 `qmd update` 增量（卷五第 30 章） | 新笔记自动进索引 |
| 第 6 月 | 检索器官稳定服务，token 大降 | 日常无感走 QMD |

### 20.6 一个验真：装完必须冒烟，别信"装好了"

呼应卷三第 15 章——QMD 装完别信"应该能用"，要真跑一次中文检索冒烟。本书的验真就是搜"心态变化 独立开发"看是否跨类命中、搜"活体知识代谢"看语义召回。两个都中（0.88/0.93 分）才算装对。**装好 ≠ 能用，能用要冒烟验。**

> **挂回**：本书架构（QMD 检索器官 + 部署纠偏）· 卷一第 0 章（省 96% token）· 卷二第 8 章（模型一致性）· 卷三第 14 章（三式）· 卷三第 15 章（冒烟验真）· A7 教训卡（三大坑）
> **下一章预告**：第 21 章装脊柱——GitHub mirror，让大脑有备份、有版本、能跨机器，且 PII 永不上云。

## 第 21 章 · 脊柱：GitHub mirror（push-only，排除 PII）

> **核心问题**：怎么让大脑不丢、能跨机器、还能托管你的技能？
> **核心命题**：GitHub private mirror 做备份/版本/跨机/技能托管，push-only critical，PII 整子树排除。

### 21.1 真实的开场：改名导致全机 git 提交挂掉

实践中踩过一个真实的脊柱事故：某次改了主机名，导致 git 的 `user.email` 变成非法的 `运营者@<hostname>.(none)`，**整机所有 repo 的自动提交全挂**——包括大脑 mirror、配置备份、技能镜像的 cron auto-push。修复方法是 `git config --global user.email/name` 设对。

这个案例说明：**脊柱看似最不起眼（平时无感），但它一断，整个持久化链就瘫了。** 备份系统的健康，要主动监控，不能等到要恢复时才发现没在备份。

### 21.2 第一性定义：脊柱的四个职责

| 职责 | 说明 |
|---|---|
| **备份** | 本地大脑全量推到 private repo |
| **版本** | git 历史 = 大脑的时间机器 |
| **跨机** | 换电脑/多设备 clone 即恢复 |
| **技能托管** | 你的 skill/脚本也托管在 GitHub |

关键约束：**push-only critical**（只推不依赖拉，本地是源，GitHub 是镜像）+ **PII 整子树 .gitignore 排除**（卷二第 11 章）。

### 21.3 反例对照：有脊柱 vs 没脊柱

| 场景 | 🔴 没 GitHub mirror | 🟢 有 mirror |
|---|---|---|
| 硬盘坏了 | 大脑全没了 | clone 即恢复 |
| 想看三个月前的版本 | 没了 | git log 时间机器 |
| 换新电脑 | 手动搬，易丢 | clone 即用 |
| PII 安全 | （无备份谈不上） | .gitignore 排除，PII 不上云 |

### 21.4 参考实现的深层细节：推前必须先 fetch + merge

实践中的脊柱铁律（从真实事故沉淀）：**force push 前必须先 fetch**（检查 origin 是否领先），且**持久化写入必须 merge 不能 overwrite**（卷二第 6 章那个 49→2 条覆盖事故）。脊柱的健康靠两条：(1) 推前先看云端状态，(2) 写入数学保证单调不减。实践中的 知识库 mirror 是 `<你的私有仓库>` private，134MB/8300 文件，.gitignore 排除敏感子树 和动态 log。

### 21.5 训练阶梯：搭好脊柱

| 时间 | 训练目标 | 自检方法 |
|---|---|---|
| 第 1 周 | 建 private repo，首次推全量（排除 PII） | git status 看不到 PII |
| 第 1 月 | 设对 git user.email/name，挂 auto-push | cron 提交正常 |
| 第 3 月 | 验证跨机：另一台 clone 能恢复 | clone 后大脑完整 |
| 第 6 月 | 脊柱健康监控（备份没断） | 定期确认 push 成功 |

### 21.6 一个延伸：GitHub Actions 可以跑代谢

脊柱不只是备份。实践中发现：**GitHub Actions（ubuntu runner）能跑通国内财经/云 API**（akshare/yfinance/阿里云 OSS 全通，1m55s 端到端）。这意味着大脑的某些代谢管道（如定时摄入、内容产出）可以从本地 launchd 迁到 GitHub Actions——尤其当本地 macOS 的 launchd 被 TCC 权限拦时（卷六第 34 章）。脊柱可以兼做"云端代谢工厂"。

> **挂回**：本书架构（GitHub 脊柱 push-only）· 卷二第 6 章（merge 不 overwrite）· 卷二第 11 章（PII 排除）· 卷六第 34 章（launchd TCC → GitHub Actions）· A7 教训卡（改名挂 git）
> **下一章预告**：第 22 章装深研器官——NotebookLM，什么时候用这个"重器官"做多源综合和音频播客。

## 第 22 章 · 深研器官：NotebookLM（多源深综 + 音频）

> **核心问题**：超出本地能力的深度研究和音频，用什么？
> **核心命题**：NotebookLM 做多源深综+播客音频，按库路由，产物回写真相源。

### 22.1 真实的开场：28 个库的路由

实践中接了 NotebookLM 28 个库，用一个 skill（NotebookLM 路由）做顶层路由——触发词（"Google笔记本/NotebookLM"）自动路由到对应库（创业四件套、量化 187 源、客户专属等）。深研器官不是"又一个搜索"，是**把一组源材料喂给一个能做深度综合 + 生成播客音频的云端大模型**。

它重、它是云、它有延迟——所以它不做日常检索（那是 QMD 的活，卷二第 8 章）。它专做"我要对这一摞材料做一次深挖/出个音频"的重活。

### 22.2 第一性定义：深研器官的两个独特能力

| 能力 | 本地（QMD/wiki）做不到的 |
|---|---|
| **多源深度综合** | 把几十个源喂进去做长篇综合研究 |
| **音频播客** | 把知识转成可听的对话式播客 |

这两个是 NotebookLM 独有的，QMD（检索片段）和 wiki（编译骨架）都给不了。

### 22.3 反例对照：用对 vs 用错深研器官

| 需求 | 🔴 用错 | 🟢 用对 |
|---|---|---|
| 找一段细节 | 开 NotebookLM 深研（杀鸡用牛刀） | QMD 秒级 |
| 对 30 个源做综合研究 | QMD（只给片段，无综合） | NotebookLM 深研 |
| 想听音频版 | 没有 | NotebookLM 播客 |

### 22.4 参考实现的深层细节：产物必须回写真相源

深研器官最容易违反的铁律是"器官产物回流真相源"（卷二第 8 章）。NotebookLM 做完研究，产物不能留在 NotebookLM（否则变第二真相源），要**回写到 Wiki 的 sources/**，再走正常消化代谢。实践中的 NotebookLM 四件套部署（含 nlm CLI、qiaomu 输入端、get-biji 兜底）就是为了让 NotebookLM 的输入输出都和真相源打通。

### 22.5 训练阶梯：接深研器官

| 时间 | 训练目标 | 自检方法 |
|---|---|---|
| 第 1 周 | 建第一个 NotebookLM 库，喂一组源 | 能做一次深度问答 |
| 第 1 月 | 生成一次音频播客 | 听到知识的音频版 |
| 第 3 月 | 多库路由（按主题分库） | 不同主题进不同库 |
| 第 6 月 | 深研产物回写真相源 | NotebookLM 产物入 Wiki sources/ |

### 22.6 一个判断：不是所有人都需要它

诚实地说：深研器官是五器官里**最可选**的一个。如果你的大脑还小、还没有"对一摞材料做深度综合"的需求，可以先不接（卷六 B3 才接它）。**别因为它酷就先装**——这正是本卷反复警告的工具拜物教。先把本体（19）、检索（20）、脊柱（21）跑稳，深研按需再说。

> **挂回**：本书架构（深研器官 + 产物回流）· 卷二第 8 章（器官产物回流真相源）· 卷六第 33 章（B3 阶段接深研）· 卷四第 25 章（别因酷而装）
> **下一章预告**：第 23 章装运营窗口——飞书，怎么把给 AI 看的真相源，变成给人/团队看的看板，且 PII 走脱敏。

## 第 23 章 · 运营窗口：飞书（看板 + 脱敏 bridge）

> **核心问题**：给 AI 优化的真相源，怎么变成给人/团队看的界面？
> **核心命题**：飞书做人面看板/团队协作/PII 业务运营，走脱敏 bridge，不当真相源。

### 23.1 真实的开场：把真相源变成给人看的运营节点

实践中建过一套飞书运营节点——真相源（给 AI 高效读）在本地，飞书是营业窗口（给人看）：看板 + 运营节点（概念卡 + 综述文档 + 协同画板）。核心设计是 **真相源 × 营业窗口**，bridge 负责中间同步。

为什么需要飞书？因为真相源（markdown + 三层 schema）是给 AI 高效读的，**不适合给人/团队浏览和协作**。飞书把知识变成人看得舒服、能协作、移动端能刷的形态。

### 23.2 第一性定义：运营窗口的三个职责

| 职责 | 说明 |
|---|---|
| **人面看板** | 把知识做成人看得爽的 dashboard |
| **团队协作** | 多人查看/评论（不是真相源，是视图） |
| **PII 业务运营** | 客户/敏感记录业务的营业界面（脱敏视图） |

铁律：飞书**不当真相源**（卷二第 8 章），它是 Wiki 的下游视图。

### 23.3 反例对照：飞书当窗口 vs 当真相源

| | 🔴 飞书当真相源 | 🟢 飞书当运营窗口 |
|---|---|---|
| 数据权威 | 双真相源打架（卷二第 6 章那个故事） | Wiki 权威，飞书是视图 |
| PII | 原始 PII 上飞书 | 脱敏 bridge，原始 PII 留本地 |
| 改数据 | 在飞书改不回流 | 在 Wiki 改，同步到飞书 |

### 23.4 参考实现的深层细节：bridge 是有方向的

实践中的飞书 bridge 是**有方向的**：Wiki → 飞书（同步知识/脱敏视图），飞书发现的修正 → 回流 Wiki（再同步出去）。绝不允许飞书成为独立的写入源。PII 走脱敏（卷二第 11 章）：飞书只见聚合/事实，原始 PII 永远留本地。实践中实测过 lark-cli 操作 Bitable/画板的一堆坑（多选字段、文档更新需关代理等），接的时候要留意。

### 23.5 训练阶梯：接运营窗口

| 时间 | 训练目标 | 自检方法 |
|---|---|---|
| 第 1 周 | 建第一个飞书看板，手动同步一批知识 | 团队能看到 |
| 第 1 月 | 建立 Wiki→飞书 单向同步 | 飞书数据来自 Wiki |
| 第 3 月 | PII 业务走脱敏 bridge | 飞书无原始 PII |
| 第 6 月 | bridge 双向（修正回流） | 飞书修正能回 Wiki |

### 23.6 一个判断：和深研器官一样，可选

飞书（运营窗口）的必要性取决于你**有没有"给别人看/团队协作/PII 业务"的需求**。纯个人大脑可以先不接。实践中接它，是因为有你的团队/生态的团队协作和客户业务需求。**你没这些需求就别接**——又一次提醒：按需，不按酷。

## 第 24 章 · 代谢引擎：让代谢自运转

> **核心问题**：怎么让摄入/消化/产出/反哺不靠手动，自己跑？
> **核心命题**：cron/hook/MCP/sub-agent 把五环代谢自动化——这是"活体"区别于"手动搬运"的字面机制。

### 24.1 真实的开场：6 条 cron 管道每天自动代谢

实践中的代谢引擎是 6 条 cron 管道（见 本书架构 + LLM Wiki 模式（Karpathy））：摄入（Get笔记/妙记自动入库）、消化（每天 12:03 源→概念）、产出-星球（7:13）、产出-视频（7:43）、反哺（周日 20:07）、仪表盘（11:57）。**你睡觉时，大脑在自己代谢。**

这就是卷一第 2 章"活体 vs 图书馆"的字面差别：图书馆要你手动搬，活体大脑自己代谢。代谢引擎就是让大脑"活"起来的那套自动化。

### 24.2 第一性定义：四种自动化机制

| 机制 | 干什么 | 例子 |
|---|---|---|
| **cron** | 定时跑 | 每天 12:03 消化源材料 |
| **hook** | 事件触发 | 摄入后自动 qmd update |
| **MCP** | AI 调用工具 | QMD/飞书的 MCP server |
| **sub-agent** | 并行干活 | 多 agent 并行抽取 |

### 24.3 反例对照：手动 vs 自动代谢

| 环 | 🔴 手动 | 🟢 自动化 |
|---|---|---|
| 摄入 | 每次手动入库 | 来源自动入 raw |
| 消化 | 手动让 AI 消化 | cron 每天消化 |
| 检索索引 | 手动 qmd embed | hook 摄入即 update |
| 反哺 | 想起来才标 | cron 周更回标 |

### 24.4 参考实现的深层细节：自动化要先手动跑顺再上

关键纪律：**别一上来就全自动化**。实践中的顺序是先手动跑顺一条管道，确认逻辑对，再挂 cron。否则一个有 bug 的脚本自动跑，会持续污染大脑（比如卷二第 6 章那个空数据覆盖事故，就是自动化脚本 fallback 出错导致的）。代谢引擎是 B4 阶段（卷六第 34 章）才上的，前三阶段先手动。

### 24.5 训练阶梯：上代谢引擎

| 时间 | 训练目标 | 自检方法 |
|---|---|---|
| 第 1 周 | 先全手动跑通五环一次 | 一条知识手动走完①→⑤ |
| 第 1 月 | 把最高频的一环（如检索索引）挂 hook | 摄入后自动 update |
| 第 3 月 | 消化/反哺挂 cron | 大脑开始自代谢 |
| 第 6 月 | 五环大部分自动化 | 睡觉时大脑在代谢 |

### 24.6 一个真实坑：macOS launchd 的 TCC 拦截

实践中实测：macOS 的 launchd 调度脚本，如果路径在 `~/Documents/` 下会被 TCC（隐私保护）拦截，必须给 /bin/bash 加完整磁盘访问权（无法自动化的 GUI 操作），或把项目搬到 `~/<project>/`，或迁到 GitHub Actions（卷四第 21 章）。代谢引擎上 cron 时这个坑必踩，知道了就不慌。

> **挂回**：本书架构（6 cron 管道）· LLM Wiki 模式（Karpathy）（ingest/query/lint 三操作）· 卷一第 3 章（五环循环）· 卷二第 6 章（自动化覆盖事故）· 卷六第 34 章（B4 上引擎）· A7 教训卡（launchd TCC）
> **下一章预告**：第 25 章是本卷的"紧箍咒"——讲完所有工具后，必须回到那个最重要的提醒：能力服务在先，工具服务能力，别陷入工具拜物教。

## 第 25 章 · 能力先于工具：别陷入工具拜物教

> **核心问题**：装了这么多工具，怎么不被工具反噬？
> **核心命题**：工具是仆人不是主人；能力（卷三六肌肉）服务在先，工具服务能力。

### 25.1 真实的开场：源文章为什么说"5 分钟"

源文章里 Andrew Levine 装 QMD"从下载到能用，他说花了 5 分钟"。Tobi 把几万行代码打包成自动安装。这些都在传递一个信号：**工具应该极简、即装即用，像 grep 一样是基础设施，不该成为你折腾的对象。**

但现实里，无数人把"建第二大脑"变成了"折腾 Obsidian 插件/调 QMD 参数/配各种自动化"——工具装了一堆，知识一篇没代谢。这就是工具拜物教：**手段变成了目的。**

### 25.2 第一性定义：工具拜物教的判定

> **工具拜物教**：花在"装/配/调工具"上的时间，远超花在"摄入/检索/代谢知识"上的时间。

判定问题：过去一周，你**用大脑代谢知识**的时间多，还是**折腾大脑工具**的时间多？后者多 = 你病了。

### 25.3 反例对照：拜物 vs 务实

| | 🔴 工具拜物 | 🟢 能力务实 |
|---|---|---|
| 装工具 | 看到酷的就装 | 只装服务某块肌肉的 |
| 时间分配 | 90% 折腾配置 | 90% 代谢知识 |
| 深研/运营器官 | 一上来全装（用不上） | 按需，B3 才接 |
| 衡量成功 | "我装了 X 个工具" | "我的知识代谢得多顺" |

### 25.4 参考实现的深层细节：每个工具都要能回答"服务哪块肌肉"

本卷介绍的每个工具，都明确对应卷三的某块肌肉：Obsidian/schema 服务摄入和组织、QMD 服务检索、GitHub 服务（持久化是验真和复利的前提）、NotebookLM 服务深度综合、飞书服务对外、代谢引擎服务整个循环的复利。**装一个工具前，先回答"它服务我哪块肌肉"——答不上来，就别装。**

### 25.5 训练阶梯：守住务实

| 时间 | 训练目标 | 自检方法 |
|---|---|---|
| 第 1 周 | 只装本体+检索（19+20），不贪多 | 两个核心器官跑通 |
| 第 1 月 | 每装一个工具回答"服务哪块肌肉" | 0 个"为酷而装"的工具 |
| 第 3 月 | 代谢知识时间 > 折腾工具时间 | 时间分配自检通过 |
| 第 6 月 | 工具隐入背景，只剩代谢 | 几乎不再折腾工具 |

### 25.6 收束：工具是为了让你忘记工具

最好的工具状态是"无感"——源文章说装完 QMD"Claude Code 调用第二大脑时会自动走 qmd，你不用做任何事"。**工具的终极目标是让你忘记它存在，专注于知识本身。** 当你不再想着"我在用什么工具"，只想着"我要代谢什么知识"，你就赢了。

> **挂回**：本书架构（五器官各服务一环）· 卷三（六块肌肉=工具要服务的对象）· 卷四第 22/23 章（深研/运营按需不按酷）· 卷一第 3 章（代谢才是目的）
> **下一卷预告**：卷五 · 何时做 · 元节奏——工具都齐了，卷五解决"按什么顺序接、接完按什么节奏养"：部署顺序（先真相源后器官）+ 日常代谢节奏（摄入/检索/维护/自动化的时刻表）。
